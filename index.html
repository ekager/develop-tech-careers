Git Some Projects

Alright, learning data structures is great and all, but when are we going to ACTUALLY MAKE SOMETHING. This is how I felt my whole first year of coding. Sorting lists, greedy algorithms, and data structures don't seem like anything software engineers actually do...
These things are the foundations that will get you to thinking like a programmer though! But let's get to the actual projects which you will need to have to find any developer job.
You can make anything your heart desires, but here are some examples to start with!

  • Your personal website! Definitely recommended as a must do. Great news is there are SO many tutorials online that you'll have a website up and running in no time. Check out the hints and tricks section for some tips!
  • A mobile app that is either a clone of an app you like or something simple like a notes app.
  • An add-on/extension for a website
  • Make a game

Make sure whatever projects you're building that you are putting the code on Github or another code sharing platform so that you can link to it on job applications! If you have no idea where to start, that's okay. There are plenty of tutorials where you can follow along and learn and build something, and then build something similar yourself! Don't reinvent the wheel and don't feel bad using templates or starting points on Github.

Hints and Tips

  • Sign up for Github! You will need an account for the next steps. If you are still a student with a .edu email sign up for the Student Developer Pack which includes some really great free trials and subscriptions.
  • Anyone can host a personal site on Github FOR FREE! Find out how here. Your site will be yourusername.github.io. Additionally you can get fairly cheap domain from a variety of companies and set it through your Github repository like this.

FAQ

The official definition is "Git is a version control system for tracking changes in computer files and coordinating work on those files among multiple people." Another way to say this is git helps you keep track of different versions of your project. Say you and I start a project and are both working on it. We would need one copy that is the official copy. Then we could each have a copy of the official copy (a fork) created by cloning the official copy that we need to keep updated with the official while we work. When we want to make some changes, we will create a new branch to work with.
Most people I talk to who are software engineers didn't necessarily choose a speciality. Everyone learns the basics and maybe you do one project where you find out that you really like front end web development, so you then take your experience and apply to jobs in that field. A lot of junior software engineering positions that you apply for won't even necessarily tell you what you'll be working on until you start, so someone will choose for you. It's good in tech to be able to adapt quickly to work on something totally new, because technology is changing all of the time. Knowing Java or Python may not be relevant in 5, 10, or 15 years, just like mobile app development wasn't a thing 10 or 15 years ago. Learning the basics well and working on a range of projects will prepare you to be constantly learning.
